### 1.1 当前结论

当前 `intent_recon_sys` 已经具备较完整的“意图识别中控”雏形：基于 FastAPI 接收会话请求，通过 Agent Registry 裁剪候选智能体，使用 LLM 输出结构化路由决策，并维护事件、计划、聊天历史和路由日志。

开源重构的核心方向不是复制 OAC 平台能力，而是将其中可复用的通用能力抽象为：

> 一个轻量级、可插拔、面向多智能体应用的意图识别与 Agent 编排框架。

开源项目名称已确认使用 `open-intent-router`。

需要特别说明：当前后端主要负责“识别意图并返回路由动作”，真实子智能体调用大多由 OAC 前端、Coze 代理或业务页面完成。开源项目应补齐通用 Agent 调用抽象，但不能把该能力误认为当前系统已完整实现。

## 2. 项目背景

`intent_recon_sys` 是 OAC 平台的中控系统，服务于银行私行财富场景。它的主要职责包括：

1. 接收统一 AI 侧边栏或子智能体窗口的用户输入。
2. 根据用户权限标签裁剪可用智能体。
3. 调用 LLM 识别用户意图，并输出结构化路由结果。
4. 决定是直接回复、澄清、打开子智能体、继续当前智能体、退出当前智能体，还是展示多步骤计划。
5. 接收子智能体执行事件，更新计划步骤，并为后续路由提供上下文。
6. 记录会话历史、事件、计划、Agent 结果和 LLM 路由日志。

现有实现强绑定 OAC 平台、飞书多维表格、Coze Bot、银行私行业务标签和示例业务智能体。开源项目需要保留通用中控能力，剥离具体业务与平台依赖，形成可独立部署、可嵌入其他应用的通用框架。

## 3. 现有系统分析

### 3.1 项目结构概览

当前系统主要目录如下：

| 目录 | 当前职责 | 开源重构判断 |
| --- | --- | --- |
| `app/api` | FastAPI HTTP 接口，包括中控路由、Agent、管理、知识库、会话接口 | `central`、`agents`、`sessions` 可复用思想；`admin` 需移除飞书强绑定；`knowledge` 应拆为可选插件 |
| `app/services` | 编排服务，包括注册表、上下文、聊天历史、事件、LLM 路由、知识库策略等 | 是核心可沉淀层，但需重命名、抽象接口、减少业务语义 |
| `app/repositories` | 内存与数据库仓储，实现 Agent、事件、结果、计划、聊天历史、路由日志等存储 | 可复用仓储分层思路；需要增加文件注册表实现与通用 Repository 协议 |
| `app/schemas` | Pydantic 请求、响应、事件、路由、Agent schema | 可复用结构化 schema 思路；字段需通用化 |
| `app/integrations` | DeepSeek/LangChain、飞书、Milvus、Embedding 等集成 | DeepSeek 可改为 LLM Provider 适配器；飞书应移出核心；Milvus/Embedding 应改为可选能力 |
| `app/db` | SQLAlchemy 模型和建表逻辑 | 可作为 DB 后端基础，但表结构需通用化 |
| `sql` | PostgreSQL 初始化和种子数据 | 建表思路可复用；种子业务数据需替换为通用示例 |
| `tests` | 路由、计划、注册表、上下文、知识库等测试 | 可复用测试场景设计；需替换业务断言和示例智能体 |
| `docs` | OAC 中控、知识库、接口和部署文档 | 可作为理解现有系统依据；不应直接进入开源核心文档 |

### 3.2 核心启动入口

当前启动入口为 `app/main.py`：

1. 创建 FastAPI 应用，标题为 `OAC Central Backend`。
2. 挂载 `admin`、`agents`、`central`、`knowledge`、`knowledge_admin`、`sessions` 路由。
3. 在 lifespan 中，如果 `REPOSITORY_BACKEND=database`，则调用 `create_all_tables()` 自动建表。
4. 提供 `/health` 健康检查。

开源重构建议：

1. 应将应用命名调整为通用名称，例如 `Open Intent Router` 或 `Intent Router`.
2. 核心路由保留 `route`、`agents`、`sessions`、`runs/events`。
3. 管理端接口应从飞书同步改为通用 Agent Registry CRUD。
4. 知识库接口应作为可选扩展，不进入 MVP 核心。

### 3.3 意图识别流程

当前主流程在 `app/api/central.py` 的 `POST /api/v1/central/route`：

1. `ChatHistoryService.hydrate_central_input()` 根据 `session_id` 补齐中控历史和当前 Agent 历史。
2. `AgentRegistryService.build_candidate_agents_for_llm()` 根据 `user_tags` 过滤候选 Agent。
3. 查询最近 Agent 执行结果、引用事件、活跃计划。
4. 可选执行知识库策略与检索，将证据放入路由上下文。
5. `ContextService.build_route_context()` 组装 LLM 上下文。
6. `LLMRouterService.route()` 调用 LLM，输出 `RouterOutput`。
7. 写入 LLM route log、conversation events、chat messages。
8. 如果返回 `plan`，写入计划仓储。

当前 `RouterOutput` 的核心字段：

```json
{
  "request_id": "req_001",
  "session_id": "sess_001",
  "route": {
    "status": "ok",
    "action": "open_agent",
    "agent_id": "daily_wecom_task",
    "message": "将为您调用企微文案生成智能体。"
  },
  "context": {
    "source": "central_chat",
    "current_agent_id": null,
    "relation": "new_task",
    "artifact_refs": []
  },
  "plan": null
}
```

可沉淀能力：

1. 候选 Agent 裁剪后再给 LLM，降低误路由风险。
2. 使用严格 Pydantic schema 约束 LLM 输出。
3. 将 `reply`、`clarify`、`open_agent`、`continue_agent`、`exit_agent`、`show_plan` 等动作标准化。
4. 用事件、历史、计划和结果构造连续上下文。
5. 路由日志可用于调试、评估和回放。

需要重构点：

1. `central`、`agent_chat` 等命名应抽象为 `router`、`conversation`、`agent_run` 等通用概念。
2. Prompt 中大量 OAC、中枢、合规质控、知识库策略等业务约束应模块化。
3. `RouterOutput.route.agent_id` 只能表达目标，缺少通用调用参数、运行模式、调用状态和调用结果协议。
4. 当前后端不统一执行 Agent，开源项目应设计可选的 `AgentInvoker` 层。

### 3.4 Agent Registry 实现方式

当前 Agent Registry schema 位于 `app/schemas/agents.py`，主要字段包括：

| 字段 | 当前含义 |
| --- | --- |
| `agent_id` | Agent 唯一标识 |
| `name` | 展示名称 |
| `description` | 功能描述 |
| `bot_id` | Coze Bot 或聊天 Bot 目标 |
| `route_path` | OAC 内部页面路径 |
| `allowed_user_tags` | 可用用户标签 |
| `domain` | 业务领域 |
| `positive_keywords` | 正向触发词 |
| `negative_keywords` | 负向触发词 |
| `required_inputs` | 必填输入描述 |
| `optional_inputs` | 可选输入描述 |
| `output_schema` | 输出字段描述 |
| `enabled` | 是否启用 |

当前数据库表为 `agent_registry`，字段与上述基本一致。当前注册表有两类来源：

1. 内存 seed：用于本地和测试。
2. PostgreSQL：生产和联调主存储。

另外当前存在飞书多维表格同步逻辑：

1. `app/integrations/feishu_bitable.py` 调用飞书 API。
2. `AgentRegistrySyncService` 将飞书字段映射为数据库 payload。
3. `app/api/admin.py` 的管理接口会同步写飞书和数据库。

开源重构建议：

1. 移除飞书作为核心注册源。
2. 数据库作为主要注册表。
3. YAML/JSON 本地配置作为兜底注册源。
4. 未来支持更多注册源时，通过 `RegistrySource` 插件扩展，不内置强绑定业务平台。

字段迁移建议：

1. `bot_id`、`route_path` 是当前业务落地字段，开源项目不保留为顶层字段。
2. `bot_id` 应迁移到 `invocation.config` 或 `provider_config`，仅在第三方平台 adapter 中解释。
3. `route_path` 应迁移到 `ui_handoff.route`，表示宿主应用接管路径。
4. `required_inputs`、`optional_inputs` 和 `output_schema` 的设计应保留，但语义升级为通用槽位和契约：
   - `required_inputs` 用于描述运行 Agent 前必须满足的槽位。
   - `optional_inputs` 用于描述可增强结果但非必需的槽位。
   - `input_schema` 用于机器可校验的输入 JSON Schema。
   - `output_schema` 用于机器可校验的输出 JSON Schema。
5. 当 Router 无法从用户输入、上下文、历史结果或 artifact 中提取必填槽位时，应返回 `clarify`，由中控追问用户，而不是直接调用 Agent。

### 3.5 子智能体调用逻辑

当前项目中没有完整的后端统一 Agent 调用执行器。当前表现为：

1. 路由输出 `open_agent` 或 `continue_agent`。
2. 前端根据 `agent_id`、`bot_id` 或 `route_path` 打开 Coze Chat、内部页面或其他能力。
3. 子智能体执行完成后，前端或业务系统通过 `/api/v1/central/events/agent` 回流结果。
4. 后端将结果写入 `agent_results`、`chat_messages`，并更新计划步骤。

这说明当前系统沉淀的是“路由决策与事件编排”，不是完整的“后端执行平台”。开源项目如果要支持通用 Agent/Tool/Workflow，应新增独立的调用抽象：

1. `AgentDefinition` 描述能力、输入输出、调用类型和运行约束。
2. `AgentInvoker` 负责执行目标 Agent。
3. `AgentRun` 记录一次执行实例。
4. `AgentEvent` 作为进度、结果、错误、澄清的统一回流协议。

### 3.6 会话上下文管理方式

当前会话上下文主要由以下数据构成：

| 数据 | 当前来源 | 用途 |
| --- | --- | --- |
| 中控聊天历史 | `chat_messages` 中 `source=central_chat` | 判断新任务、追问、引用 |
| Agent 聊天历史 | `chat_messages` 中 `source=agent_chat` 且按 `agent_id` 过滤 | 判断继续当前 Agent 或切换 Agent |
| 最近事件 | `conversation_events` | 支持页面导航、路由触发、Agent 事件引用 |
| 最近结果 | `agent_results` | 支持“刚才那条”“上一版”等引用 |
| 活跃计划 | `plans`、`plan_steps` | 支持多步骤任务推进 |
| 前端上下文 | `frontend_context` | 支持当前活跃 Agent、页面状态等 |
| 知识库证据 | 可选 `knowledge_context` | 支持基于证据回复或路由 |

当前 SQL 中存在 `session_states` 表，但主流程未明显使用。开源项目可保留“轻量事件溯源 + 可选 session summary/state”的双模型：

1. MVP 阶段以消息、事件、结果、计划为主。
2. 后续再引入 `session_state`，用于意图栈、槽位、摘要和活跃运行态。

### 3.7 配置文件和数据库相关逻辑

当前配置集中在 `app/config.py`，主要配置包括：

1. DeepSeek 模型和 API Key。
2. PostgreSQL 和 Redis URL。
3. `repository_backend=memory/database`。
4. 飞书 API 配置。
5. Router history 限制。
6. 知识库、Milvus、Embedding、上传等配置。

开源重构建议：

1. 保留 Pydantic Settings 模式。
2. 将 LLM Provider 配置抽象为 `ROUTER_LLM_PROVIDER`、`ROUTER_LLM_MODEL`、`ROUTER_LLM_BASE_URL`。
3. 将注册表配置抽象为 `REGISTRY_BACKEND=database/file/hybrid`。
4. 将 Agent 调用配置拆到注册表字段，不放入全局配置。
5. 将知识库、Milvus、飞书、OAC header 从核心配置中移除或迁入插件。

### 3.8 业务强绑定模块

当前系统的主要业务绑定点包括：

| 绑定点 | 当前表现 | 开源处理建议 |
| --- | --- | --- |
| OAC 平台 | README、FastAPI title、接口文档、路径、部署域名、OAC header | 核心项目统一改为 Host Application；OAC 相关内容只出现在迁移 adapter 文档中 |
| 银行私行财富场景 | 文档中的理财经理、运营版、展业版、企微、合规、海报等 | 替换为通用示例 Agent，不进入核心逻辑 |
| Coze | `bot_id`、文档中的 Coze Chat/SSE、`coze_workflow` | 抽象为 `provider_platform` 类型 Agent Adapter |
| 飞书 | 飞书多维表格同步、中文字段映射、双写逻辑 | 不进入开源核心，也不作为 MVP 插件；如未来需要，只能由社区或宿主应用自定义 registry source |
| DeepSeek | 默认 LLM 客户端和 prompt 中模型描述 | 抽象为 OpenAI-compatible 或任意 LLM Provider |
| Milvus/阿里 Embedding | 共享知识库检索和索引 | 作为可选 Evidence/RAG 插件 |
| 权限标签 | 硬编码 `运营版`、`展业版` | 改为通用 `visibility_rules` / `access_policy` |
| 路由目标 | `bot_id` 与 `route_path` 二选一 | 改为通用 `invocation` 与 `ui_handoff` |

## 4. 开源重构目标

### 4.1 总体目标

将 OAC 中控系统中的通用能力抽象为一个可独立使用的意图识别与 Agent 编排框架，支持开发者在自己的应用中：

1. 注册多种 Agent、Tool 或 Workflow。
2. 基于用户输入、会话历史和上下文识别意图。
3. 输出结构化路由决策。
4. 可选地调用目标 Agent。
5. 接收 Agent 执行事件和结果。
6. 支持多步骤计划、用户确认和任务推进。
7. 提供数据库主注册表和本地配置兜底。

### 4.2 可复用能力

当前系统中建议沉淀为开源核心的能力：

| 当前模块 | 可复用能力 | 重构方向 |
| --- | --- | --- |
| `LLMRouterService` | 结构化 LLM 路由 | 抽象 LLM Client，支持多 Provider |
| `RouterOutput` | 路由输出协议 | 保留动作模型，增加置信度、原因、调用参数 |
| `AgentRegistryService` | 候选 Agent 裁剪 | 增加通用 access policy、capabilities、invocation |
| `ContextService` | 上下文构造 | 抽象为 `RouteContextBuilder` |
| `ChatHistoryService` | 会话历史补齐 | 通用化 message source 和 participant |
| `EventService` | Agent 事件回流 | 扩展为 Agent Run 事件模型 |
| `PlanRepository` | 多步骤计划管理 | 扩展依赖、参数映射、步骤结果引用 |
| repository 分层 | memory/database 两套存储 | 增加 file registry、接口协议、迁移机制 |
| route logs | 路由可观测性 | 增加评估、回放和审计字段 |
| tests | 路由场景测试 | 替换为通用示例测试 |

### 4.3 需要删除、替换或抽象的能力

| 当前内容 | 处理方式 |
| --- | --- |
| 飞书多维表格同步 | 从核心删除；后续作为插件示例 |
| `bot_id`、`route_path` | 替换为 `invocation`、`ui_handoff`、`provider_config` |
| OAC 文档、部署域名、header | 删除或放入独立迁移说明 |
| 银行业务 seed agent | 替换为通用 sample，如 weather、ticket、summarizer、mock |
| 硬编码权限标签 | 替换为通用 access policy |
| DeepSeek 专属配置 | 替换为 LLM provider adapter |
| Milvus/Embedding 知识库 | 移出 MVP 核心，作为 Evidence Provider 插件 |
| 后台知识文件管理 | 不进入开源意图识别 MVP |

## 5. 项目定位

### 5.1 一句话定位

一个轻量级、可插拔、面向多智能体应用的意图识别与 Agent 编排框架。

### 5.2 适合解决的问题

1. 一个应用中存在多个 Agent、Tool、Workflow，需要统一识别用户意图并选择目标能力。
2. 需要根据用户权限、租户、渠道、场景动态裁剪候选 Agent。
3. 需要在主会话和子 Agent 会话之间保持上下文连续性。
4. 需要支持多步骤任务计划，例如“先生成文案，再审核，再发布”。
5. 需要将 Agent 执行结果回流到统一中控，由中控决定下一步。
6. 需要本地开发可用的 Mock Agent 和文件注册表兜底。

### 5.3 不适合解决的问题

1. 不替代 LangGraph、Temporal、Airflow 等复杂工作流执行引擎。
2. 不提供领域业务知识库的完整建设方案。
3. 不内置某个垂直行业的 Prompt、规则和合规模型。
4. 不作为通用聊天 UI 框架。
5. 不作为传统低代码平台或 Bot 平台的完整替代。
6. 不承诺开箱即用连接所有第三方 Agent 平台，MVP 只提供清晰扩展接口。

### 5.4 面向开发者

1. 正在构建多 Agent 应用的独立开发者。
2. 希望在现有 SaaS、CRM、运营后台、知识库系统中接入 AI 中控的团队。
3. 需要将多个内部 API、LLM Agent、工作流统一暴露给自然语言入口的后端开发者。
4. 希望从“简单 prompt 分类器”升级为“带上下文、权限、事件和计划的路由器”的应用开发者。

### 5.5 与相关技术的关系

| 对象 | 关系 |
| --- | --- |
| Prompt 分类器 | 本项目可以使用 Prompt/LLM 分类，但不仅分类，还维护候选能力、上下文、事件、计划和调用协议 |
| 传统规则引擎 | 本项目可支持规则预过滤，但核心是自然语言意图识别和 Agent 编排，不追求替代 Drools 等规则系统 |
| LangChain | 可作为 LLM Provider 或 Agent Adapter 之一，本项目不强依赖 LangChain |
| LangGraph | 可作为 Workflow 类型 Agent 的执行目标；本项目负责发现、路由、调用和事件汇聚 |
| Dify/Coze/FastGPT | 可作为第三方平台 Agent Provider，通过 adapter 接入 |
| API Gateway | 本项目不是通用 API 网关；只处理意图、路由、上下文和 Agent 运行态 |

## 6. 用户角色与使用场景

### 6.1 用户角色

| 角色 | 说明 | 关注点 |
| --- | --- | --- |
| Framework Maintainer | 开源项目维护者 | 核心抽象稳定性、插件边界、测试覆盖 |
| App Developer | 应用开发者 | 快速注册 Agent、接入自身应用、扩展调用方式 |
| Agent Developer | 子智能体开发者 | 输入输出协议、事件回流、调试工具 |
| Operator / End User | 使用宿主应用的最终用户 | 能否自然表达需求、连续完成任务 |
| Admin User | 管理者 | Agent 上下线、权限、注册表维护、日志排查 |

### 6.2 典型使用场景

#### 场景一：多 Agent 自然语言入口

用户在宿主应用中输入“帮我整理这份会议纪要，再创建一个待办”。系统识别为多步骤任务：

1. 调用 summarizer Agent。
2. 将摘要结果传给 task creator Agent。
3. 返回计划，等待用户确认。
4. 执行并汇总结果。

#### 场景二：继续当前 Agent

用户在某个 Agent 会话中输入“再短一点”。系统结合 `current_agent_id` 和历史结果，判断为 `continue_agent`，不重新路由到其他 Agent。

#### 场景三：切换 Agent 并引用上一步结果

用户输入“把刚才那版发到工单系统”。系统从最近结果中提取 artifact 引用，切换到 ticket Agent，并传入上一步产物。

#### 场景四：数据库注册表不可用时本地兜底

生产环境数据库暂时不可用，服务启动时加载 `agents.yaml` 作为只读注册表，保证基础 Agent 可用，同时将健康状态标记为 degraded。

#### 场景五：本地开发与测试

开发者使用 Mock Agent 和本地 YAML 注册表，不依赖外部平台，即可验证意图识别、路由和计划推进。

### 7.2 Agent Registry API

MVP 需要提供：

| 接口 | 用途 |
| --- | --- |
| `GET /api/v1/agents` | 查询 Agent 列表 |
| `GET /api/v1/agents/{agent_id}` | 查询单个 Agent |
| `POST /api/v1/admin/agents` | 新增或更新 Agent |
| `PATCH /api/v1/admin/agents/{agent_id}/enabled` | 启用或停用 Agent |
| `DELETE /api/v1/admin/agents/{agent_id}` | 删除 Agent |
| `POST /api/v1/agents/available` | 根据用户上下文返回可用 Agent |
| `POST /api/v1/admin/registry/reload` | 重新加载文件注册表或刷新缓存 |

MVP 不提供飞书同步接口。

### 7.3 Agent 调用 API

开源项目应支持两种模式：

1. `route_only`：只返回路由决策，由宿主应用执行 Agent。
2. `route_and_invoke`：路由后由框架调用目标 Agent。

建议接口：

| 接口 | 用途 |
| --- | --- |
| `POST /api/v1/invoke` | 明确调用某个 Agent |
| `POST /api/v1/route-and-invoke` | 路由并调用 |
| `GET /api/v1/runs/{run_id}` | 查询 Agent Run |
| `POST /api/v1/runs/{run_id}/events` | 回流 Agent 事件 |
| `POST /api/v1/events/agent` | 兼容当前系统的事件回流入口 |

### 7.4 会话与历史

MVP 应支持：

1. 按 `session_id` 查询主会话历史。
2. 按 `session_id + agent_id + agent_session_id` 查询 Agent 历史。
3. 写入用户消息、Router 消息、Agent 消息。
4. 限制注入 LLM 的历史条数。
5. 支持清理或归档会话。

### 7.5 计划管理

MVP 应支持：

1. LLM 返回多步骤计划。
2. 计划保存为 `pending`。
3. 用户确认后变为 `running`。
4. Agent 事件回流后更新步骤状态。
5. 当前步骤完成后推进下一步骤。
6. 失败、阻塞、澄清状态可中断计划。

### 7.6 可观测性

MVP 应支持：

1. 记录候选 Agent 列表。
2. 记录路由输入摘要。
3. 记录结构化输出。
4. 记录解析错误和 LLM 调用错误。
5. 记录 Agent Run、事件和执行耗时。

### 7.7 Evidence Provider 与固定问映射

开源项目应保留当前知识库检索能力，但将其抽象为首个可选 `Evidence Provider` 插件，而不是核心强依赖。

该插件需要支持两类能力：

1. 证据检索：为路由和回复提供可追溯证据。
2. 固定问映射：当用户问题与已配置的固定问模板匹配时，直接产出意图提示、候选 Agent 或路由优先级。

固定问映射的目标是支持“用户问题命中固定问 -> 归因到固定问所映射的意图”。它应发生在常规 LLM 路由前，可作为 `intent_hint`、`candidate_agent_ids` 或 `route_override` 注入后续路由流程。

## 8. 非功能需求

### 8.1 可插拔性

1. LLM Provider 可替换。
2. Registry Source 可替换。
3. Agent Invoker 可扩展。
4. Storage Backend 可替换。
5. Prompt 模板可配置。

### 8.2 可靠性

1. 数据库不可用时可使用本地文件注册表兜底。
2. LLM 调用失败时返回结构化错误，不抛出未处理异常。
3. Agent 调用超时时记录失败事件。
4. 事件回流应具备幂等性，重复 `event_id` 不重复推进计划。

### 8.3 安全性

1. 管理接口必须有鉴权机制。
2. Agent 注册表中敏感认证信息不得明文暴露给普通查询接口。
3. HTTP Agent 调用应支持超时、域名 allowlist 和 header 脱敏日志。
4. 本地函数 Agent 需要明确注册，不允许任意动态 import 用户输入路径。

### 8.4 可测试性

1. 提供 Mock LLM Client。
2. 提供 Mock Agent Invoker。
3. 提供内存仓储。
4. 提供 YAML 示例注册表。
5. 核心路由、计划推进、事件幂等、注册表兜底必须有单元测试。

### 8.5 性能要求

MVP 建议目标：

| 指标 | 建议目标 |
| --- | --- |
| Registry 可用 Agent 查询 | p95 < 100ms，缓存命中时 |
| Router API 非 LLM 部分 | p95 < 200ms |
| LLM 调用超时 | 默认 20s，可配置 |
| Agent HTTP 调用超时 | 默认 30s，可按 Agent 配置 |
| 单实例内存模式并发 | 支持本地开发和测试，不承诺生产持久化 |

## 10. Agent 抽象设计

### 10.1 核心概念

| 概念 | 说明 |
| --- | --- |
| Agent | 可被路由和调用的能力单元，可以是远程服务、本地函数、工作流或第三方平台 Bot |
| Tool | 通常是无状态、单次调用的能力，可作为 Agent 的一种轻量类型 |
| Workflow | 多节点、有状态或长耗时的流程，可作为 Agent 的一种执行类型 |
| Agent Definition | Agent 的注册表定义 |
| Agent Invocation | 一次调用请求 |
| Agent Run | 一次调用实例，包含状态、事件、结果 |
| Agent Event | 运行过程中的进度、结果、失败、澄清等事件 |
| Artifact | Agent 产物引用，例如文档、图片、结构化结果、外部资源 URL |

### 10.2 Agent 类型

MVP 及后续应支持以下类型：

| 类型 | MVP | 说明 |
| --- | --- | --- |
| `http` | 是 | 调用 HTTP API 类型 Agent |
| `local_function` | 是 | 调用本地注册函数，用于轻量工具和本地开发 |
| `mock` | 是 | 固定返回或规则返回，用于测试 |
| `workflow` | 后续 | LangChain、LangGraph 或其他工作流 |
| `provider_platform` | 后续 | Coze、Dify、FastGPT 等第三方平台 |
| `ui_handoff` | 是 | 只返回宿主应用打开页面或接管执行，不由后端直接调用 |

### 10.3 Agent Definition 结构

建议通用注册表字段：

```json
{
  "agent_id": "summarizer",
  "name": "Text Summarizer",
  "description": "Summarize long text into concise notes.",
  "version": "1.0.0",
  "enabled": true,
  "type": "http",
  "capabilities": ["summarize", "extract_key_points"],
  "domain": "productivity",
  "tags": ["text", "summary"],
  "trigger": {
    "positive_examples": ["summarize this", "make this shorter"],
    "negative_examples": ["translate this"],
    "keywords": ["summary", "summarize"]
  },
  "access_policy": {
    "allow_roles": ["operator"],
    "allow_groups": ["default"],
    "allow_tenants": ["*"]
  },
  "input_schema": {
    "type": "object",
    "required": ["text"],
    "properties": {
      "text": {"type": "string"}
    }
  },
  "output_schema": {
    "type": "object",
    "properties": {
      "summary": {"type": "string"}
    }
  },
  "invocation": {
    "type": "http",
    "config": {
      "method": "POST",
      "url": "http://localhost:9001/summarize",
      "timeout_seconds": 30
    }
  },
  "ui_handoff": {
    "mode": "none"
  },
  "metadata": {
    "owner": "demo",
    "source": "agents.yaml"
  }
```

### 10.6 槽位与执行前校验

`required_inputs`、`optional_inputs`、`input_schema` 和 `output_schema` 应作为 Agent 抽象的一等字段保留。

执行前流程：

1. Router 根据 Agent Definition 读取 `required_inputs`。
2. 从用户输入、会话历史、最近 Agent 结果、artifact refs 和前端上下文中抽取槽位。
3. 使用 `input_schema` 校验已抽取输入。
4. 如果必填槽位缺失，返回 `clarify`，并在 `message` 中追问最小必要信息。
5. 如果必填槽位齐全，生成 `invocation.input`，进入 `route_and_invoke` 或返回给宿主应用执行。
6. Agent 返回后使用 `output_schema` 做结构校验；校验失败时记录 `agent_result.status=failed` 或 `invalid_output`。

示例：

```json
{
  "required_inputs": ["customer_segment", "task_goal"],
  "optional_inputs": ["tone", "deadline"],
  "slot_policy": {
    "on_missing_required": "clarify",
    "max_clarification_questions": 2
  }
}
```

### 10.7 Agent Invoker 接口

建议 Python 抽象：

```python
class AgentInvoker(Protocol):
    async def invoke(
        self,
        definition: AgentDefinition,
        invocation: AgentInvocation,
    ) -> AgentInvocationResult:
        ...
```

不同类型由不同 adapter 实现：

| Adapter | 职责 |
| --- | --- |
| `HttpAgentInvoker` | 调用远程 HTTP API |
| `LocalFunctionInvoker` | 调用应用内注册函数 |
| `MockAgentInvoker` | 返回测试结果 |
| `UiHandoffInvoker` | 不执行，仅生成宿主应用接管信息 |
| `LangGraphWorkflowInvoker` | 后续接入 LangGraph |
| `ProviderPlatformInvoker` | 后续接入 Coze/Dify/FastGPT |

### 10.8 扩展机制

1. 使用 `type` 字段决定调用 adapter。
2. 使用 `invocation.config` 保存不同 adapter 的配置。
3. 核心只校验公共字段；特定配置由 adapter 校验。
4. 支持通过 Python entry points 或显式注册函数扩展 adapter。
5. 插件不得修改核心 RouterOutput schema，只能扩展 `metadata` 或 `provider_payload`。

## 11. Agent Registry 设计

### 11.1 Registry 目标

Agent Registry 是 Router 的能力目录。它不仅用于展示 Agent，还用于：

1. 裁剪当前用户可用 Agent。
2. 给 LLM 提供候选 Agent 摘要。
3. 描述 Agent 输入输出约束。
4. 描述 Agent 调用方式。
5. 控制 Agent 上下线和版本。

### 11.2 数据库字段设计

建议数据库表 `agent_definitions`：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | bigint | 是 | 自增主键 |
| `agent_id` | varchar(128) | 是 | 唯一业务 ID |
| `name` | varchar(200) | 是 | Agent 名称 |
| `description` | text | 是 | 给用户和 LLM 的能力描述 |
| `version` | varchar(64) | 否 | Agent 版本 |
| `type` | varchar(64) | 是 | `http/local_function/mock/workflow/provider_platform/ui_handoff` |
| `enabled` | boolean | 是 | 是否启用 |
| `domain` | varchar(200) | 否 | 领域或分类 |
| `capabilities_text` | json/text | 是 | 能力标签数组 |
| `trigger_text` | json/text | 是 | 触发词、正负例 |
| `access_policy_text` | json/text | 是 | 权限、租户、角色策略 |
| `input_schema_text` | json/text | 是 | JSON Schema |

### 11.4 数据库和本地文件共存机制

MVP 建议支持三种模式：

| 模式 | 说明 |
| --- | --- |
| `database` | 仅使用数据库，数据库不可用则启动失败 |
| `file` | 仅使用本地 YAML/JSON，适合本地开发 |
| `hybrid` | 数据库为主，本地文件为兜底 |

`hybrid` 加载规则：

1. 启动时先尝试连接数据库。
2. 如果数据库可用，从数据库加载所有启用 Agent。
3. 如果数据库不可用，加载本地文件注册表，并将服务状态标记为 `degraded`。
4. 如果数据库可用但返回空列表，可按配置决定是否合并文件示例：
   - `REGISTRY_FILE_FALLBACK_ON_EMPTY=true` 时加载文件。
   - 默认不自动合并，避免意外启用示例 Agent。
5. 运行时数据库恢复后，可通过 `/api/v1/admin/registry/reload` 手动刷新。

### 11.5 冲突处理

当多个来源出现相同 `agent_id`：

1. `database` 模式下只认数据库。
2. `file` 模式下同一文件内重复 `agent_id` 应启动失败。
3. `hybrid` 模式下数据库可用时数据库优先。
4. `hybrid` 模式下数据库不可用时文件生效。
5. 未来多源合并时使用 `source_priority` 明确优先级，不做隐式覆盖。

### 11.6 启动时加载流程

```text
应用启动
  -> 读取 Settings
  -> 初始化 RegistryManager
  -> 根据 REGISTRY_BACKEND 选择 source
  -> 校验 Agent Definition schema
  -> 构建内存缓存
  -> 暴露 /health 和 /ready 状态
```

健康状态建议：

| 状态 | 含义 |
| --- | --- |
| `ok` | 主注册源可用 |
| `degraded` | 主注册源不可用，正在使用文件兜底 |
| `error` | 无可用注册源或配置校验失败 |

### 11.7 后续扩展注册源

后续可扩展：

1. Git 仓库注册源。
2. S3/Object Storage 注册源。
3. Consul/Etcd 注册源。
4. 第三方平台同步源。
5. 管理后台或 SaaS 控制台。

核心要求：新增注册源只能实现 `RegistrySource` 接口，不得把具体平台字段写入核心 schema。

## 12. 意图识别与路由流程

### 12.1 路由输入格式

建议输入 schema：

```json
{
  "request_id": "req_001",
  "session_id": "sess_001",
  "source": "host_chat",
  "user": {
    "id": "user_001",
    "roles": [],
    "groups": [],
    "attributes": {}
  },
  "input": {
    "type": "text",
    "text": "用户输入",
    "attachments": []
  },
  "current_agent": {
    "agent_id": "summarizer",
    "run_id": "run_001",
    "agent_session_id": "agent_sess_001"
  },
  "event_id": null,
  "plan_id": null,
  "step_id": null,
  "frontend_context": {}
}
```

### 12.2 候选 Agent 生成

候选生成流程：

1. 从 Registry 读取启用 Agent。
2. 根据用户 `roles/groups/tenant/attributes` 执行 access policy。
3. 根据 source 和当前上下文过滤不可用 Agent。
4. 可选根据关键词、embedding 或规则做预排序。
5. 将裁剪后的 Agent 摘要传给 LLM。

在进入 LLM 路由前，系统应先尝试固定问匹配：

1. 对用户输入做规范化处理，例如去除多余空白、统一大小写、同义词归一。
2. 调用可选 Evidence Provider 的 fixed-question matcher。
3. 如果命中固定问，将其映射结果写入 `intent_hint`、`candidate_agent_ids` 或 `route_override`。
4. 如果固定问配置为强路由，可跳过 LLM，直接返回对应意图或 Agent。
5. 如果固定问配置为弱提示，则继续进入 LLM 路由，但将命中信息作为高优先级上下文。

传给 LLM 的候选 Agent 不应包含密钥、HTTP header、内部 URL 等敏感配置，只包含：

```json
{
  "agent_id": "summarizer",
  "name": "Text Summarizer",
  "description": "Summarize long text into concise notes.",
  "domain": "productivity",
  "capabilities": ["summarize"],
  "trigger": {
    "keywords": ["summary"],
    "positive_examples": ["summarize this text"],
    "negative_examples": ["translate this text"]
  },
  "required_inputs": ["text"]
}
```

### 12.3 路由决策数据结构

建议路由决策结构：

```json
{
  "decision": {
    "status": "ok",
    "action": "open_agent",
    "target_agent_id": "summarizer",
    "confidence": 0.91,
    "reason": "用户明确要求总结文本",
    "message": "将为您调用文本总结能力。"
  },
  "context": {
    "relation": "new_task",
    "current_agent_id": null,
    "artifact_refs": [],
    "candidate_agent_ids": ["summarizer", "translator"]
  },
  "plan": null,
  "invocation": {
    "mode": "deferred",
    "agent_id": "summarizer",
    "input": {
      "text": "用户输入"
    }
  }
}
```

动作枚举建议保留现有经验：

| action | 说明 |
| --- | --- |
| `reply` | Router 直接回复 |
| `clarify` | 需要用户补充信息 |
| `open_agent` | 打开或启动目标 Agent |
| `continue_agent` | 当前 Agent 继续处理 |
| `exit_agent` | 退出当前 Agent |
| `show_plan` | 展示多步骤计划 |
| `unsupported` | 不支持该请求 |
| `silent` | 不产生用户可见回复 |

关系枚举：

| relation | 说明 |
| --- | --- |
| `new_task` | 新任务 |
| `continue_current` | 继续当前 Agent |
| `switch_agent` | 切换到另一个 Agent |
| `exit_agent` | 退出当前 Agent |
| `multi_task` | 多步骤任务 |
| `unsupported` | 不支持 |

### 12.4 路由流程

```text
接收 RouteRequest
  -> 校验 source/current_agent/event/plan 字段
  -> 补齐会话历史
  -> 加载可用 Agent
  -> 无候选时返回 unsupported
  -> 构造 RouteContext
  -> 调用 LLM Router
  -> 校验 RouterOutput
  -> 记录 route log
  -> 记录 conversation events
  -> 写入 chat messages
  -> 如有 plan，保存计划
  -> 如启用 route_and_invoke，创建 Agent Run 或等待用户确认
  -> 返回结构化响应
```

### 12.5 路由器实现策略

MVP 推荐使用 LLM 结构化输出作为主路由器，并保留规则前后处理：

1. 前处理：权限裁剪、候选数量限制、当前 Agent 校验。
2. LLM 路由：根据 Prompt 和 JSON schema 输出决策。
3. 后处理：校验 `target_agent_id` 必须在候选集中。
4. 后处理：`show_plan` 必须包含 plan，非 `show_plan` 不得包含 plan。
5. 后处理：`continue_agent` 默认目标必须是 `current_agent.agent_id`。
6. 后处理：输出不合法时返回 `error/reply` 或触发 fallback。

## 17. 风险、决策与待办

### 17.1 主要风险

| 风险 | 影响 | 建议 |
| --- | --- | --- |
| 当前系统未后端统一调用 Agent | 开源项目若承诺过大，MVP 范围会失控 | 明确 MVP 只实现基础 invoker，复杂平台 adapter 后续做 |
| LLM 路由不稳定 | 误路由或输出不合法 | 保留严格 schema、候选校验、后处理和评估集 |
| Registry 字段过度设计 | 开发成本上升 | MVP 字段满足 http/local/mock/ui_handoff，复杂字段放 metadata |
| 文件兜底误启用示例 Agent | 生产风险 | hybrid 模式默认数据库优先，不自动合并文件示例 |
| 本地函数 Agent 安全问题 | 任意代码执行风险 | 只允许显式注册函数，不允许用户输入动态加载 |
| 第三方平台适配差异大 | 核心被平台字段污染 | 使用 adapter + provider_config，核心不感知平台私有字段 |

### 17.2 已确认决策

1. 开源项目名称确认为 `open-intent-router`。
2. 提供后端 `route_and_invoke`，但 MVP 只实现基础 invoker。
3. 默认 LLM Provider 采用 OpenAI-compatible，同时保留 DeepSeek 示例。
4. 数据库首选 PostgreSQL，本地开发默认 SQLite。
5. Agent Registry 需要完整 CRUD。
6. 需要提供 OAC 迁移 adapter 文档，但不进入开源核心。
7. 保留当前知识库检索作为首个可选 Evidence Provider 插件，并支持固定问映射到对应意图。

### 17.3 非阻塞不确定点

以下问题不会阻塞需求分析，但会影响后续设计细节：

1. 当前 OAC 前端的 `CENTRAL_AGENT_ROUTE_MAP` 只是子智能体路由映射，不影响开源核心设计。
2. 当前子智能体真实执行链路不完全在 `intent_recon_sys` 后端内，因此开源执行层属于新增设计，该方向已确认。
3. 当前 `session_states` 表存在但主流程未明显使用，MVP 暂时不启用。
4. 当前知识库能力较完整，后续作为 Evidence Provider 插件保留；除检索知识外，也支持用户问题命中固定问后映射到对应意图。
